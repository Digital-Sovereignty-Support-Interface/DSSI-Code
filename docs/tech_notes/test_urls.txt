における、それぞれの具体的な活用法を解説します。

1. example.com (IANA管理)
ステータス: 永久に使える

DSSIでの用途: 「HTTP警告」の動作確認

このサイトは http:// と https:// の両方でアクセス可能です。

テスト手順:

http://example.com にアクセスする。

安心ポイント: RFC 2606で予約されているため、将来誰かに買収されて変なサイトになるリスクがゼロです。

2. httpbin.org (HTTP Request & Response Service)
ステータス: 使える（が、Docker推奨）

DSSIでの用途: 「通信内容の可視化」

現在はPostman社がホストしています。

注意点: ブラウザ拡張機能開発においては、ページ（HTML）ではなくJSONを返すため、DOM要素（入力フォーム）の検知テストには向きません。

活用: 将来的にDSSIが「裏でどんな通信が走っているか」を解析する機能を作る際、リクエストヘッダの確認などに使えます。

推奨: 公開サーバーは遅いことがあるため、開発時はDockerでローカルに立てるのが一般的です (docker run -p 80:80 kennethreitz/httpbin)。

3. badssl.com (Chromiumチーム等が維持)
ステータス: DSSI開発における「最強の実験場」

DSSIでの用途: 「HTTPSインスペクターの全パターン検証」

このサイトは、DSSIのPhase 1.5（プロトコル検査）をテストするために存在すると言っても過言ではありません。

テストできること:

expired.badssl.com: 「期限切れ証明書」の警告テスト。

wrong.host.badssl.com: 「ドメイン名不一致」のテスト。

self-signed.badssl.com: 「オレオレ証明書」のテスト。

実装への示唆: DSSIが単に「HTTPSならヨシ」とするのではなく、「HTTPSだけど証明書が腐っている」場合をどう判定するか、このサイトを使えば深く検証できます。
